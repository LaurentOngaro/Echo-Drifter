# v0.1.0.0_Analyse du Proto & Feuille de route corrective

## Contexte

Ce document analyse le premier prototype généré en vibe coding (Kilo Code + MiniMax M3) à partir des specs Echo Drifter, et propose une feuille de route corrective ciblée pour rapprocher le résultat du concept original.

Il ne remet pas en cause l'architecture globale définie dans `v0.1.0-mvp-phases.md`, qui reste valide et utilisable telle quelle.

---

## Ce que le proto v0.1 a réussi

Malgré un résultat encore loin du "feel" voulu, plusieurs éléments sont déjà en place et réutilisables :

- Pipeline Vite + TypeScript + Three.js fonctionnel et stable.
- Boucle de jeu active avec update par frame.
- HUD DOM présent : titre, boutons layers (DRONE/PULSE/BASS/PAD/ARP/LEAD), jauge de dissonance, bouton RESTART.
- Présence d'orbes rondes dans la scène - la géométrie de base est juste.
- Système audio procédural branché et déclenchable.
- Structure de dossiers cohérente avec l'architecture cible.

Ces éléments sont des acquis. Ils ne sont pas à jeter, seulement à affiner.

---

## Ce qui dysfonctionne dans le proto v0.1

### Problème 1 - Rendu visuel "3D cheap"

**Symptôme** : l'image donne une impression de 3D générique, pas de l'univers sensoriel synthwave organique défini dans les specs.

**Causes probables** :

- Matériaux Three.js non contraints : `MeshStandardMaterial` avec éclairage dynamique produit un look "3D maquette" non souhaité.
- Absence de contraintes de camera rig : perspective trop prononcée, pas de vue "flat/2D-ish".
- Absence de glows, halos, gradients doux : les orbes sont des sphères sans profondeur visuelle.
- Absence de règles sur la motion : pas d'easing explicite sur les pulsations.

**Impact** : le jeu ne ressemble pas à un univers "rétrofuturiste lumineux et organique", il ressemble à un test WebGL.

### Problème 2 - Musique procédurale désagréable

**Symptôme** : les sons générés sont discordants, mécaniques, désagréables à l'oreille.

**Causes probables** :

- Pas de gamme fixée : les oscillateurs génèrent des fréquences libres ou aléatoires.
- Timbres bruts : oscillateurs square/sawtooth non filtrés en haute fréquence.
- Densité trop forte dès le début : plusieurs voix actives simultanément sans cohérence harmonique.
- Pas de règles d'intervalle : aucune contrainte sur les notes utilisables.

**Impact** : le lien "gameplay → musique → émotion" est cassé dès l'ouverture.

### Problème 3 - Le plan M3 est un plan d'ingénieur, pas de direction artistique

**Constat** : `v0.1.0-mvp-phases.md` est excellent pour la structure de code, mais ne contient pas de garde-fous esthétiques suffisants. L'agent IA comble ce vide avec des choix par défaut génériques.

**Impact** : sans brief artistique précis, le vibe coding produit systématiquement un résultat "fonctionnel mais générique".

## Feuille de route corrective

L'objectif n'est pas de refaire tout le proto, mais d'ajouter deux mini-briefs pour contraindre les prochaines passes de vibe coding vers une vraie identité.

### Étape 1 - Freezer le proto v0.1 (immédiat)

Créer un tag ou commit clair pour garder cet état.

```bash
git add -A
git commit -m "chore: freeze proto v0.1 - baseline pre-art-direction"
```

Cela permet de revenir en arrière sans risque si une passe corrective casse quelque chose.

### Étape 2 - Créer `_Helpers/docs/echo-drifter-audio-notes.md`

Ce document fixe les règles de musicalité inviolables pour la génération audio procédurale.
Il doit être lu par Kilo avant toute modification du dossier `src/audio/`.

Contenu minimum :

**Tonalité MVP** : A mineur pentatonique uniquement.
Notes autorisées : A2, C3, D3, E3, G3 (et leurs octaves supérieures).
Aucune note hors gamme au MVP.

**Timbres autorisés** :

- Drone : 2 oscillateurs sine filtrés (root + quinte), lowpass à 300–600 Hz.
- Pulse : kick procédural (sine rapide 60→20 Hz) + hat très doux (noise filtré 4 kHz, très court).
- Pad : oscillateur triangle ou sine, attack lente (1–2 sec), release longue.
- Arp : sine ou triangle, notes de la gamme seulement, vitesse 0.25–1 note/sec.
- Lead : sine ou triangle, une phrase de 4 notes courte et répétée.

**Interdits stricts** :

- Pas d'oscillateur square brut à haute fréquence.
- Pas de noise continu non filtré.
- Pas de fréquences aléatoires (Math.random() sur les Hz est interdit).
- Pas de plusieurs voix sur des hauteurs non harmoniques simultanées.

**Densité** :

- Démarrage : drone seul pendant 2–3 secondes, puis pulse rejoint.
- Chaque collectible n'ajoute qu'un seul layer.
- Maximum 3 layers simultanés avant le fragment 4.

**Dissonance** (anomalies) :

- Effet court (< 3 sec).
- Implémenté par detune léger (+20 cents max) + baisse du gain master.
- Se résout progressivement (fadeIn sur 1–2 sec).

### Étape 3 - Créer `_Helpers/docs/echo-drifter-visual-notes.md`

Ce document fixe les règles esthétiques pour la présentation Three.js.
Il doit être lu par Kilo avant toute modification de `src/presentation/`.

Contenu minimum :

**Palette** :

- Fond : `#0a0612` (violet très sombre, presque noir).
- Joueur : `#3fd6ff` (cyan doux) ou `#c084fc` (violet clair).
- Collectibles : `#ff80bf` (rose/magenta doux) ou `#ffd580` (ambre doux).
- Anomalies : `#ff4040` légèrement désaturé, jamais de rouge pur agressif.
- Glow : toujours plus saturé que la base, alpha 0.3–0.6 max.

**Matériaux autorisés** :

- `MeshBasicMaterial` ou `MeshStandardMaterial` sans specular agressif.
- Emissive activé sur les orbes (pas de rendu "plastique mat").
- Pas de `MeshPhongMaterial` avec shininess > 30.

**Géométries autorisées** :

- `SphereGeometry` (orbes joueur, collectibles).
- `TorusGeometry` (anneaux de guidance, aura).
- `PlaneGeometry` plat pour les halos / glows billboard.
- Aucun cube, aucun cône, aucun icosahèdre, aucun wirframe angulaire.

**Caméra** :

- `PerspectiveCamera` avec FOV 50–65°, pas plus.
- Position fixe en Z (5–10 unités) pour donner un effet "quasi-orthographique".
- Pas de rotation caméra dramatique ni de dolly rapide.
- Follow doux avec damping > 0.85 (lent à rattraper le joueur).

**Motion** :

- Pulsation des orbes : fréquence 0.5–1.5 Hz, amplitude scale ±5% maximum.
- Pas de rotation rapide (omega < 0.3 rad/s).
- Toutes les transitions de scale/couleur via interpolation linéaire ou ease-out.
- Collecte : ripple scale 0→2, opacity 1→0, durée 500–800 ms.

**Effets autorisés** :

- Glow billboard (plane avec texture radiale ou shader simple).
- Trail derrière le joueur (pool de petites sphères avec opacity décroissante).
- Pas de post-processing complexe au MVP (pas de bloom full-screen).

### Étape 4 - Passe corrective audio (Phase 5 revisitée)

Avec le brief audio en place, redonner les contraintes à Kilo :

> Lis `_Helpers/docs/echo-drifter-audio-notes.md`. Rewrite `src/audio/synth.ts` et `src/audio/layerController.ts` pour respecter strictement ces règles. Ne modifie pas l'architecture `audioDirector`/`musicStateMachine`. Liste les changements avant d'éditer.

Critère de validation : le drone + pulse démarrent, ils sonnent "agréables et cohérents", sans bruit discordant.

### Étape 5 - Passe corrective visuelle (Présentation revisitée)

Avec le brief visuel en place, redonner les contraintes à Kilo :

> Lis `_Helpers/docs/echo-drifter-visual-notes.md`. Refactorise `src/presentation/meshes.ts` et `src/presentation/scene.ts` pour ne produire que des orbes/anneaux doux avec les matériaux et la palette définis. Ne touche pas aux systèmes gameplay ni audio. Liste les changements avant d'éditer.

Critère de validation : la scène ressemble à un univers "rétrofuturiste doux, couleurs cohérentes, pulsation visible et agréable".

### Étape 6 - Reprendre les phases MVP dans l'ordre

Une fois les deux passes correctrices validées visuellement et sonore, reprendre le plan `v0.1.0-mvp-phases.md` à la Phase 2 (drift controller) en mode vibe coding normal.

L'ordre recommandé :

1. Phase 2 - Drift controller & camera rig.
2. Phase 3 - Collectibles & guidance.
3. Phase 4 - Anomalies.
4. Phase 5 - Audio director complet.
5. Phase 6 - HUD, restart, polish.

## Leçon structurante pour la suite

Le vibe coding avec un agent IA est très puissant pour l'architecture et la plomberie.
Il est faible par défaut pour la direction artistique et la qualité musicale.

La règle à retenir pour Echo Drifter :

- Toute nouvelle session Kilo qui touche à `src/audio/` doit relire `echo-drifter-audio-notes.md`.
- Toute nouvelle session Kilo qui touche à `src/presentation/` doit relire `echo-drifter-visual-notes.md`.

Ces deux fichiers sont les "garde-fous du goût" que l'agent seul ne peut pas inférer.
